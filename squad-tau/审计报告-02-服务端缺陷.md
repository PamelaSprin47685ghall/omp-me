# Squad-Tau 功能审计报告 — 服务端缺陷

## 高 (High)

### S-H-01: `validateNodes()` 返回值被忽略

**文件**: `server/dag-execute.js:7`  
**实际行为**: 
```js
validateNodes(nodes);
const layers = topologicalSort(nodes);
```
`validateNodes` 返回 `{ valid, errors }` 但返回值未检查。即使节点列表有重复 ID 或无效依赖，代码继续执行拓扑排序，在 `topologicalSort` 中以更模糊的错误信息失败。

**修复**: 检查返回值并抛出有意义错误：
```js
const { valid, errors } = validateNodes(nodes);
if (!valid) throw new Error(`Invalid plan: ${errors.join(', ')}`);
```

### S-H-02: `topologicalSort` 静默去重同 ID 节点

**文件**: `server/dag-sort.js:10`  
**实际行为**: `const nodeMap = new Map(nodes.map((n) => [n.id, n]))` — 当节点 ID 重复时，Map 只保留最后一个，前面的节点被静默丢弃。与 `validateNodes` 的重复 ID 检测配合时问题不大，但 topologicalSort 本身应抛错而非静默去重。

**修复**: 在 Map 构建前检查重复并抛错。

### S-H-03: session 事件缺少 `retryCount` 字段

**文件**: `server/run-worker.js:55-60`, `server/run-reviewer.js:35-41`, `server/run-node.js`  
**PRD 要求**: §5.5 `session:start` 事件必须包含 `retryCount?` 字段  
**实际行为**: Worker 和 Reviewer 的 `session:start` 事件均未包含 `retryCount`。前端 SessionTree 显示 `R<retry>-<role>` 标签，但没有 retryCount 数据，只能默认显示 `R1-Worker`。

**影响**: 重试轮次信息在 UI 中永远显示为 R1，用户无法区分当前是第几轮。

### S-H-04: `handleSessionStateChange` 只设置 `.phase`，不设置 `.status`

**文件**: `client/session-reducer.js:20-23`  
**实际行为**:
```js
function handleSessionStateChange(state, payload) {
    const { sessionId, phase } = payload;
    const sessions = new Map(state.sessions);
    const session = sessions.get(sessionId);
    if (session) sessions.set(sessionId, { ...session, phase });
    return { ...state, sessions };
}
```
`session:state` 事件只更新 `.phase`，不更新 `.status`。但 `session:end` 事件设置 `.status` 为 `completed/aborted/error`。在 `session:state` 事件中（如 `phase: 'reviewing'`），session 的 `.status` 仍然保持初始值 `'active'`。`MainContent.jsx` 的 `InputSection` 通过 `['completed', 'aborted', 'error'].includes(activeSession.status)` 判断是否禁用输入框——这依赖 `.status` 正确设置。

**影响**: Session 状态变更时 `.status` 字段停留在 `'active'`，即使阶段已变了。但输入框禁用逻辑实际检查的是 `.status`，可能不受影响因为 `session:end` 事件会设置正确的结束状态。不过如果 `session:state` 事件想传达 `phase: 'error'`，此处 `.status` 不会更新。

### S-H-05: `buildWorkerSessionOptions` 过滤了不存在的工具名

**文件**: `server/session-options.js:28-30`  
**实际行为**:
```js
const activeTools = (ctx?.session?.getActiveToolNames?.() ?? pi?.getActiveTools?.())
    ?.filter((t) => t !== 'delegate_squad');
```
实际注册的全局工具名是 `delegate`（非 `delegate_squad`），过滤器是空操作。这不会导致 bug（不会多过滤），但意图是防止 Worker 节点再次调用 delegate，当前未实现该意图。

**影响**: Worker/Reviewer 子会话理论上可以调用 `delegate` 工具创建新的 squad 嵌套，这违反了设计意图。

### S-H-06: `session:user_message` 广播缺少 `parentId` 字段

**文件**: `server/ws-handler.js:53-58`  
**PRD 要求**: §5.5 `session:message` payload 包含可选 `parentId` 字段  
**实际行为**: 用户消息广播中硬编码 `messageId: \`user-${Date.now()}\``，但没有 `parentId` 字段。虽然 `parentId` 是可选的，但其缺失意味着消息无法与触发它的 tool_call 关联。

## 中 (Medium)

### S-M-01: `session-events.js` 事件处理器无 try-catch

**文件**: `server/session-events.js:17-27`  
**实际行为**: `subscribeToSessionEvents` 订阅的回调中，`handleMessageUpdate`/`handleToolStart`/`handleToolEnd`/`handleMessageEnd` 均无 try-catch 包装。如果任何处理器抛出异常，整个 session 事件订阅中断，后续消息流停止。

**影响**: 一个格式异常的消息（如 `event.message` 为 null）会导致当前节点静默停止接收更新。

### S-M-02: Worker prompt 发送前未检查 abort 信号

**文件**: `server/run-worker.js:handleFirstReturn`  
**实际行为**: `modelPool.acquire` 后立即 `session.prompt(workerPrompt)` 而未检查 `childAbort.signal.aborted`。如果 abort 信号在 acquire 期间到达（acquire 返回 null 后），代码仍会尝试对空 session prompt。

**注意**: 当前 acquire 返回 null 后 `runWorker` 会 try-catch 处理，但 null modelSlot 的 session 创建路径可能抛出不明确的错误。

### S-M-03: `dag-concurrency.js` 发射未文档化事件

**文件**: `server/dag-concurrency.js`  
**实际行为**: 发射 `squad:node_start` 和 `squad:node_end` 事件，但 PRD §5.4 事件协议中未定义这两个事件。客户端忽略未知事件类型（不会崩溃），但增加了协议漂移风险。

### S-M-04: `outer_review_result` 事件在拒绝时双重发射

**文件**: `server/outer-review.js:138-143` + `server/submit-plan.js:82`  
**实际行为**: `processOuterVerdict` 在 `outer-review.js` 中发射了一次 `squad:outer_review_result`。然后当 verdict 为 rejected 时，`submit-plan.js` 的 `runOuterReviewLoop` 也发射了一次 `squad:outer_review_result`。客户端会收到两次 rejected 事件。

**影响**: 前端如果对 `outer_review_result` 事件做计数或动画，会出现闪烁或重复提示。

## 低 (Low)

### S-L-01: `server-lifecycle.js` stopServer 不发送 `connection:close` 事件

**文件**: `server/server-lifecycle.js`  
**PRD 要求**: §5.3 定义了 `connection:close` 事件用于优雅通知  
**实际行为**: `stopServer` 直接 `wss.close()` 关闭连接，不发送 `connection:close` 事件。客户端通过 `onclose` 检测断连然后重连，功能不受影响但违反 PRD 协议。

### S-L-02: `model-pool.js` release() 查找逻辑可能匹配错误槽位

**文件**: `server/model-pool.js:58`  
**实际行为**: 当传入的 slot 对象没有 `_slot` 引用时（理论上不会，因为 acquire 总是设置），fallback 按序匹配 `provider + modelId + inUse`。在多个相同配置槽位场景下，可能释放错误的槽位。

### S-L-03: `node_state` 事件在 FAILED/BLOCKED 状态缺少 `retryCount`

**文件**: `server/dag-execute.js:47,80`  
**实际行为**: `handleAbortedLayer` 和 `handleBlockedNodes` 发射的 `squad:node_state` 事件不包含 `retryCount` 字段，与 `run-node.js` 中 APPROVED/REJECTED 发射的字段不一致。
