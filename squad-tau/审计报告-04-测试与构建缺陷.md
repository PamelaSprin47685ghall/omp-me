# Squad-Tau 功能审计报告 — 测试/构建/基础设施缺陷

## 严重 (Critical)

### T-C-01: 核心 DAG 节点编排器 `run-node.js` 零测试

**文件**: `server/run-node.js`（3.0KB，核心模块）  
**影响**: `runNode` 是整个 squad 执行的核心编排函数——管理 Worker→Confirm→Reviewer 的完整生命周期和重试循环。它是系统中最重要的模块之一，但没有任何单元测试覆盖。`runWorker` 和 `runReviewer` 只有 prompt 构建器的测试（`run-worker.test.js` 和 `run-reviewer.test.js` 仅测 `buildWorkerPrompt` / `buildReviewerPrompt`），不覆盖异步执行循环、空轮次保护、会话注册/注销、事件发射、模型槽位 acquire/release 等关键路径。

**建议**: 为 `runNode` 编写集成级测试，至少覆盖：
- 正常路径：Worker 提交 → Confirm 通过 → Reviewer 通过
- 重试路径：Reviewer 驳回 → 重新 Worker
- Abort 信号传播
- 空轮次超时
- 模型槽位 acquire/release 配对

## 高 (High)

### T-H-01: 18 个集成/e2e 测试文件全部 `.skip.js`，永不运行

**文件**: `test/integration/*.skip.js`（6个）+ `test/e2e/*.skip.js`（12个）  
**影响**: 关键端到端流程（WebSocket 通信、完整 squad 流程、DAG 执行、混沌测试）完全未被 CI 覆盖。PRD §8.4 和 §8.5 定义的端到端测试策略形同虚设。这些 `.skip.js` 文件不会被 `bun test` 自动发现。

### T-H-02: 9 个单元测试使用 `fs.readFileSync` 源码模式匹配

**文件**: `test/unit/round3-audit.test.js`, `round4-gaps.test.js`, `bugs-4-7.test.js`, `final-bugs.test.js`, `dead-code.test.js`, `duplicate-code.test.js`, `deps.test.js`, `vite-middleware.test.js`, `http-server.test.js`  
**影响**: 这些测试用 `fs.readFileSync` 读取源文件内容然后用正则/字符串匹配验证代码是否存在。它们测试代码文本而非运行时行为——如果函数存在但逻辑错误，测试仍通过。这不是有意义的测试。

### T-H-03: mock-pi.js 有 8 个空操作桩，掩盖真实行为

**文件**: `test/helpers/mock-pi.js`  
**空操作**: `sendMessage`, `sendUserMessage`, `setModel`, `setActiveTools`, `setSessionName`, `getActiveTools`, `setThinkingLevel`, `setStatus`  
**影响**: 任何代码调用这些方法都不会报错，即使调用签名错误（如传多余参数）也不会被检测到。特别是 `sendUserMessage`——它是最关键的用户交互方法，但在 mock 中是空函数。

### T-H-04: `bun test` worker 复用导致 mock 模块泄漏

**文件**: 已记录在 `PROB.md`  
**影响**: `mock.module('@oh-my-pi/resolve-pi')` 在 `squad-flow-setup.js` 中注册后，worker 复用时泄漏到其他测试文件。6 个集成测试一起跑时失败，必须分开执行。根因是 `mock.module` 无法清除 ESM 模块缓存。

## 中 (Medium)

### T-M-01: 2 个测试文件使用 vitest 而非 bun:test

**文件**: `test/unit/empty-turns.test.js`, `test/unit/retry-logic.test.js`  
**影响**: `bun test` 通过 vitest 兼容层运行这些文件，但行为可能不一致（如 mock 清理、定时器模拟）。与项目其余 30+ 个 `bun:test` 测试不统一。

### T-M-02: `model-pool-config.test.js` 写入真实 CONFIG_PATH

**文件**: `test/unit/model-pool-config.test.js`  
**影响**: 测试直接用 `process.cwd()/.omp/models.toml` 作为配置路径，意味着测试运行后会在工作目录留下测试文件。应使用临时目录。

## 低 (Low)

### T-L-01: 4 个服务端模块无任何测试

**文件**: `server/ws-heartbeat.js`, `server/plugin-state.js`, `server/session-options.js`  
**影响**: `ws-heartbeat.js` 管理 30s ping/60s 超时，心跳逻辑未测试。`plugin-state.js` 管理全局可变状态，无断言验证。`session-options.js` 构建会话配置包括模型映射和工具过滤，直接影响执行路径。

### T-L-02: `squad-engine.js` 仅有模式匹配测试

**文件**: 服务端主入口 7.8KB，涉及命令注册、FSM 管理、会话创建等核心逻辑。唯一相关测试是 `round3-audit.test.js` 中的字符串匹配，不测试运行时行为。

## 构建/基础设施

### B-01: `package.json` 根级 Vite 脚本缺失配置

**文件**: `package.json`  
**实际行为**: `dev`/`build`/`preview` 脚本调用 `vite` 但未指定 `--config`。根目录无 `vite.config.js`（配置在 `client/vite.config.js`）。运行 `bun run dev` 会报"No vite config found"。实际服务端通过 `server/vite-setup.js` 用 Node API 创建 Vite 服务器，绕过了这些脚本。

### B-02: `@vitejs/plugin-react` 在 devDependencies 中未使用

**文件**: `package.json`  
**影响**: 列在 devDependencies 中，但从未被任何文件 import。JSX 转换通过 Vite 的 esbuild 内置处理。

### B-03: `shim.mjs` 缺失

**文件**: 项目根目录  
**PRD 参考**: REF-05 规定标准插件结构包含 `shim.mjs`。当前项目只有 `index.js` 入口，通过 `package.json` 的 `main` 字段加载。功能上可以工作，但不符合插件标准。

### B-04: `@oh-my-pi/shim` 锁文件残留

**文件**: `bun.lock`  
**影响**: `bun.lock` 引用 `@oh-my-pi/shim`（file:../shim-packages/pi-shim），但目标目录不存在。`node_modules/@oh-my-pi/shim/` 是断开的符号链接。不影响当前运行，但 `bun install` 可能失败。
