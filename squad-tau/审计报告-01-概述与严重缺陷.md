# Squad-Tau 功能审计报告 — 概述与严重缺陷

**审计日期**: 2026-05-11  
**审计范围**: `server/`、`client/`、`test/`、配置文件  
**审计重点**: 功能正确性与缺陷，非安全性  

---

## 1. 审计概要

| 类别 | 严重 | 高 | 中 | 低 | 合计 |
|------|------|----|----|----|------|
| 服务端 | 4 | 6 | 4 | 3 | 17 |
| 客户端 | 3 | 2 | 4 | 3 | 12 |
| 测试/构建 | 1 | 4 | 2 | 2 | 9 |
| **合计** | **8** | **12** | **10** | **8** | **38** |

## 2. 架构评价

**优点**：
- 模块拆分清晰，文件平均 80-150 行，符合 ≤200 行约束
- 事件总线 + 命名空间设计干净，WebSocket 桥接逻辑解耦
- 状态机纯函数实现，易于测试
- 模型池 acquire/release 队列模式正确，pendingDelete 语义完整
- 客户端 useWebSocket 的 onEventRef 模式正确避免闭包陈旧

**系统性风险**：
- 全局注册的 `return` 工具依赖 session-registry 按 sessionId 路由，但 OMP 创建子 session 时 sessionId 是文件路径而非自然数，与 PRD §5.3 矛盾
- 乐观消息（optimistic message）与服务器回声之间无去重机制
- 客户端 thinking 块渲染完全依赖 delta 增量，message.content 中的 thinking 数据被忽略

## 3. 严重缺陷（Critical）

### C-01: `syncModelPoolFromConfig` Map 键去重破坏重复槽位

**文件**: `server/model-pool-config.js:41-67`  
**PRD 要求**: §6.2 明确指出"同一配置行可重复（例如 3 行相同的 worker 配置 = 3 并发）"  
**实际行为**: `syncModelPoolFromConfig` 使用 `provider|modelId|role` 组合作为 Map 键。当用户配置 3 个相同 worker 槽位时，Map 只保留一个，另外两个被静默丢弃。

```js
const key = `${entry.provider}|${entry.modelId}|${entry.role}`;
if (!oldMap.has(key)) {
    modelPool.addSlot(entry);  // 只添加一次
}
```

**影响**: 用户配置的并发度被意外降低。3 个相同 worker 槽位只产生 1 个并发，其余 acquire 调用将排队等待。

### C-02: `createConfirmPrompt` 只处理对象格式 review_criteria

**文件**: `server/run-worker-prompt.js:57-69`  
**PRD 要求**: §3.2 `review_criteria` 格式为 `string | string[] | {name, description}[]`  
**实际行为**: `buildConfirmPrompt` 只处理 `criteria.length > 0` 的数组，假设元素是 `{name, description}` 对象。但 `formatReviewCriteria`（worker prompt 中）正确处理了三种格式。Confirm prompt 中：

```js
if (criteria.length > 0) {
    criteriaSection = '\n' + criteria.map((c) => `- ${c.name}: ${c.description}`).join('\n');
}
```

当 `review_criteria` 是 string 或 `string[]` 时，`c.name` 和 `c.description` 都是 `undefined`，Self-Confirm 阶段的评审标准渲染为 `- undefined: undefined`。

**影响**: LLM 在 Self-Confirm 阶段收到损坏的评审标准提示词，自审质量严重下降。

### C-03: Worker 和 Confirm 阶段各自独立 MAX_EMPTY_TURNS 计数器

**文件**: `server/run-worker.js:19-20,33`（引用 `empty-turns.js`）  
**PRD 要求**: §3.2 明确指出"两阶段共享 MAX_EMPTY_TURNS = 20"  
**实际行为**: `handleFirstReturn` 创建独立的 `createCounter(MAX_EMPTY_TURNS)` 实例，`handleSecondReturn` 也创建独立实例。Worker 阶段可达 20 空轮次，Confirm 阶段也可达 20 空轮次，总计 40 而非 20。

**影响**: 违反 PRD 规格。节点可在 Worker+Confirm 共 40 个空轮次后才终止，浪费 token。

### C-04: DAG 执行失败不发送 `squad:complete` 事件

**文件**: `server/dag-execute.js`, `server/submit-plan.js:97-103`  
**PRD 要求**: §5.4 定义了 `squad:complete` 事件包含所有节点结果  
**实际行为**: `executeDAG` 的异常在 `submit-plan.js` 的 `runDelegate` 中被 catch 并 throw 新 Error，但没有任何 `squad:complete` 或 `squad:abort` 事件发出。客户端无法知道 squad 已失败——UI 永远停留在"运行中"状态。

```js
// submit-plan.js runDelegate
try {
    const results = await executeDAG({ nodes, ctx, pi, signal, eventBus, modelPool });
    return await finalize({ results, ...deps });
} catch (error) {
    throw new Error(`DAG execution failed: ${error.message}`);
    // 未 emit squad:complete 或 squad:abort
}
```

**影响**: DAG 执行失败时前端永远不知道 squad 已结束，必须手动刷新。

### C-05: 客户端 ErrorBanner 使用不存在的 `.error` 字段

**文件**: `client/components/ErrorBanner.jsx:23-25`  
**实际行为**:
```js
const reason = blocked.length > 0
    ? blocked[0].error || 'Node blocked'
    : failed[0].error || 'Node failed';
```

节点状态对象（来自 `squad:node_state`）只有 `summary` 字段，没有 `error` 字段。`dag-execute.js` 中 FAILED/BLOCKED 节点的结果只有 `{ nodeId, status, summary, affectedFiles }`。而 `run-node.js` 的 catch 块确实会 emit `error: error.message`，但这仅限于 `runNode` 抛出的情况，`executeDAG` 的 `handleAbortedLayer` 和 `handleBlockedNodes` 不带 error 字段。

**影响**: ErrorBanner 大多数场景显示 fallback 文本 "Node blocked"/"Node failed" 而非具体原因，用户无法了解失败详情。

### C-06: 客户端乐观消息与服务器回声造成重复渲染

**文件**: `client/App.jsx:117`, `client/components/MessageInput.jsx`  
**实际行为**: `MessageInput.handleSend` 调用 `onOptimisticMessage` 立即创建 `role:'user'` 消息，然后服务端广播 `session:message` 回声再次添加相同消息（不同 messageId）。`session-reducer.js` 的 `handleSessionMessage` 按 `messageId` 查重，但乐观消息的 ID 是 `opt_timestamp_counter`，服务器回声的 ID 是 `user-timestamp`，两者不同，不会被去重。

**影响**: 用户发送一条消息，UI 显示两条相同内容。这是用户可见的视觉问题。

### C-07: 客户端 thinking 块仅从 delta 渲染，message.content 中的 thinking 被丢弃

**文件**: `client/components/MessageItem.jsx:28-30`  
**实际行为**:
```js
const thinkingContent = msgDeltas.filter(d => d.delta.type === 'thinking_delta')
    .map(d => d.delta.text).join('');
```

thinking 内容只从当前消息的增量流式数据中提取。如果用户切换会话或重新连接，历史消息的 `message.content` 中的 `thinking` 块（来自 `session:message` 的 `handleSessionEnd` 或 `handleSessionMessage`）不会被渲染，因为没有增量数据。

**影响**: 重新连接或切换回历史会话时，所有 thinking 内容消失。

### C-08: DAGView 点击节点切换会话不生效

**文件**: `client/components/DAGView.jsx:58-64`  
**实际行为**: `useMermaidRender` 用 `stateKey = nodeStateKey(nodeList)` 作为渲染守卫。当 `activeNodeId` 变化但节点状态没变时，`stateKey` 相同，`useEffect` 提前返回 `if (stateKey === renderedKeyRef.current) return;`，图表不重新渲染。虽然 `renderDiagram` 的 `useCallback` 依赖了 `activeNodeId`，但它只在 `stateKey` 变化时被调用。

**影响**: 用户点击 DAG 节点，节点样式（圆括号 vs 方括号）不变，无法视觉反馈当前选中节点。
